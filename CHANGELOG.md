## Conventional Commits Format

This project follows the [Conventional Commits](https://www.conventionalcommits.org/) specification:

- `feat(scope): description` - New feature (minor version bump)
- `fix(scope): description` - Bug fix (patch version bump)
- `perf(scope): description` - Performance improvement (patch version bump)
- `docs(scope): description` - Documentation changes (no release)
- `style(scope): description` - Code style changes (no release)
- `refactor(scope): description` - Code refactoring (no release)
- `test(scope): description` - Test changes (no release)
- `chore(scope): description` - Chore/maintenance (no release)

Breaking changes should include `BREAKING CHANGE:` in the footer to trigger a major version bump.
