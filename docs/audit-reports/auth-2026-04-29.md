## Root Cause Analysis — ALTA: passkey tenantId mancante

### BUG-1: generateRegistrationOptions senza tenantId

```typescript
// PRIMA (VULN — OWASP A01:2025):
const user = await this.prisma.user.findUnique({
  where: { id: userId },  // ← nessun tenantId
});

// DOPO (SAFE):
const user = await this.prisma.user.findFirst({
  where: { id: userId, tenantId },  // ← tenant isolation
});
```

**Impatto:** Un attaccante con un userId valido di altro tenant poteva innescare WebAuthn registration per utenti non propri.
**Traceability:** OWASP A01:2025 (Broken Access Control), CLAUDE.md NON-NEGOTIABLE.

### BUG-2: listPasskeys / deletePasskey senza tenantId

```typescript
// PRIMA (VULN):
where: { userId }  // Passkey model ha no tenantId diretto

// DOPO (SAFE — Prisma nested relation filter):
where: { userId, user: { tenantId } }
```

**Note:** Il modello `Passkey` non ha colonna `tenantId` propria — il tenant è derivato tramite relazione `user`. La soluzione corretta usa il filtro nested Prisma `user: { tenantId }`.

### CEILING ACCETTATO: verifyAuthentication credential lookup

```typescript
// RIMASTO (CEILING — WebAuthn protocol):
const passkey = await this.prisma.passkey.findFirst({
  where: { credentialId },  // ← no tenantId per design
});
// Verifica tenant avviene post-lookup (line 168):
if (!passkey.user.isActive || !passkey.user.tenant?.isActive) { ... }
```

**Motivazione:** I credential ID WebAuthn sono globalmente unici (64 byte random, W3C spec §6.1). La ricerca deve essere globale per design. Il tenant viene verificato post-lookup controllando attività user+tenant.
