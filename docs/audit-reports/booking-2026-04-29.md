## Root Cause Analysis (BLOCCANTI)

### BUG-1: idempotencyKey senza tenantId (GDPR A01 — OWASP)
```typescript
// PRIMA (VULN):
this.prisma.booking.findUnique({ where: { idempotencyKey } })
// Se un attaccante conosce/indovina la idempotencyKey di un altro tenant,
// riceve il booking di quel tenant → data leak cross-tenant.

// DOPO (SAFE):
this.prisma.booking.findFirst({ where: { idempotencyKey, tenantId } })
```
**Impatto:** GDPR Art.32 violation, OWASP A01:2021 Broken Access Control.

### BUG-2: bulkConfirm update senza tenantId
```typescript
// PRIMA (VULN):
this.prisma.booking.update({ where: { id }, data: { status: 'CONFIRMED' } })
// update() bypassa RLS — nessun contesto tenant settato.

// DOPO (SAFE):
this.prisma.booking.updateMany({ where: { id, tenantId }, data: { status: 'CONFIRMED' } })
```
**Impatto:** Cross-tenant data mutation (basso rischio pratico dato il findFirst check precedente, ma violazione architetturale).

### BUG-3: reserveSlot con `id: 'temp'` (runtime crash)
```typescript
// PRIMA (BROKEN):
const bookingEvent = await tx.bookingEvent.create({
  booking: { connect: { id: 'temp' } }  // ← Nessun record con id='temp' → Prisma throws
});
// Il metodo reserveSlot() NON ha mai funzionato in produzione.

// DOPO (SAFE):
// 1. Update slot → BOOKED
// 2. Create booking con ID reale
// 3. Create bookingEvent con booking.id reale
```
**Impatto:** Funzionalità reserveSlot completamente inoperante in produzione. Unit test non lo rileva perché bookingEvent.create è mockato.

## Confronto stato dell'arte 2026

✅ In linea:
- Advisory lock + SERIALIZABLE per race condition prevention
- State machine `validateTransition()` su ogni cambio status
- EncryptionService per PII in decryptCustomerInBooking
- JwtAuthGuard + RolesGuard su tutti gli endpoint
- tenantId in ogni query (dopo fix)
- Paginazione su findAll

⚠️ Indietro:
- Nessun rate limiting su endpoint booking (standard 2026: throttle 10 req/min per IP)
- Business logic nel controller (statusColorMap, calendar mapping)
- N+1 decrypt in findAll (accettabile ma non ottimale)
- `bulkConfirm` loop sequenziale (standard: Promise.allSettled con concurrencyLimit)

❌ Mancante:
- CQRS/Event Sourcing per audit completo (standard fintech 2026)
- Cache Redis per slot availability (query frequente, dati semi-statici)
- Idempotenza per reschedule e cancel (solo createBooking ce l'ha)
- Correlation ID propagato nei log
- eIDAS 2.0: no EUDI wallet integration (deadline Nov 2027)
