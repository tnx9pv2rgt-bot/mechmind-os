## Root Cause Analysis (BLOCCANTI)

### RCA-1: receiveInbound non idempotente

**Problema:** Twilio garantisce at-least-once delivery — i webhook vengono ritentati se il server risponde con status ≥ 500 o timeout. Il metodo `receiveInbound` non verifica se `twilioSid` è già presente prima di creare il messaggio.

**Fix minima:**
```typescript
async receiveInbound(phoneHash: string, body: string, twilioSid?: string): Promise<unknown> {
  // Idempotency check
  if (twilioSid) {
    const existing = await this.prisma.smsMessage.findFirst({ where: { twilioSid } });
    if (existing) return existing;
  }
  // ... resto del metodo invariato
}
```

**Fix strutturale:** Aggiungere `UNIQUE` constraint su `smsMessage.twilioSid` nel schema Prisma + `upsert` invece di `create`.

### RCA-2: TWILIO_AUTH_TOKEN opzionale in produzione

**Problema:** Il codice accetta richieste senza autenticazione se `TWILIO_AUTH_TOKEN` non è configurato. In produzione, questa configurazione mancante rende l'endpoint webhook completamente aperto.

**Fix:**
```typescript
const authToken = this.configService.get<string>('TWILIO_AUTH_TOKEN');
if (!authToken) {
  if (process.env.NODE_ENV === 'production') {
    throw new InternalServerErrorException('TWILIO_AUTH_TOKEN not configured');
  }
  this.logger.warn('TWILIO_AUTH_TOKEN not configured — skipping signature verification (dev only)');
} else {
  // ... verifica firma
}
```
