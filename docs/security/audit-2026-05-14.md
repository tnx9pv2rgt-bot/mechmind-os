## MEDIUM (4.0-6.9)

### 4. Email in Plaintext Logs (GDPR Art.32 violation)

| Metric       | Value                                                                       |
| ------------ | --------------------------------------------------------------------------- |
| **ID**       | PII-LOG-001                                                                 |
| **File**     | `backend/src/auth/services/login-throttle.service.ts:61`                    |
| **Line**     | `this.logger.warn(\`Login failure #${newCount} for ${email} from ${ip}\`);` |
| **CVSS**     | 5.3 — CVSS:3.1/AV:N/AC:L/PR:N/UI:N/S:U/C:L/I:N/A:N                          |
| **CWE**      | CWE-532: Insertion of Sensitive Information into Log File                   |
| **GDPR**     | Art.32 (integrity and confidentiality of personal data)                     |
| **Severity** | Medium (logged to backend logs only, not external)                          |

**Context**: Email logged during failed login attempts. Logs may be aggregated
to central logging (CloudWatch, Sentry, etc.), exposing email addresses.

**Fix**: Mask email or use hash:

```typescript
// Before
this.logger.warn(`Login failure #${newCount} for ${email} from ${ip}`);

// After
const emailHash = createHash('sha256')
  .update(email)
  .digest('hex')
  .substring(0, 8);
this.logger.warn(
  `Login failure #${newCount} for user ${emailHash}*** from ${ip}`
);
```

**Effort**: 1 line change.

| **Line** | `const vehicle = estimate.vehicleId ? await this.prisma.vehicle.findUnique({ where: { id: estimate.vehicleId } }) : null;`                                                                                      |
| **CVSS** | 5.7 — CVSS:3.1/AV:N/AC:L/PR:L/UI:N/S:U/C:L/I:L/A:N                                                                                                                                                              |

### 7. Resend Webhook Signature Verification - Missing Timestamp Validation

| Metric   | Value                                                                                                                                                                       |
| -------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **ID**   | WEBHOOK-SIGNATURE-001                                                                                                                                                       |
| **File** | `backend/src/notifications/controllers/webhook.controller.ts:355-384`                                                                                                       |
| **Line** | Lines 355-384 (verifyResendSignature)                                                                                                                                       |
| **CVSS** | 4.8 — CVSS:3.1/AV:N/AC:L/PR:N/UI:R/S:C/C:L/I:L/A:N                                                                                                                          |
| **CWE**  | CWE-208: Observable Timing Discrepancy                                                                                                                                      |
| **Gap**  | Resend webhook signature verified via HMAC, but NO timestamp freshness check (replay protection missing). Resend expects timestamp validation to prevent replayed webhooks. |

**Fix**: Add timestamp validation (optional but recommended per Resend docs):

```typescript
private verifyResendSignature(
  payload: ResendWebhookEvent,
  signature: string,
  timestamp: string,
): boolean {
  // ... existing HMAC verification ...

  // Add timestamp freshness check (5 min window)
  const ts = parseInt(timestamp, 10);
  const now = Math.floor(Date.now() / 1000);
  if (Math.abs(now - ts) > 300) {
    this.logger.warn('Resend webhook timestamp outside 5-min window — possible replay');
    return false;
  }

  return crypto.timingSafeEqual(Buffer.from(signature), Buffer.from(expectedSignature));
}
```

**Effort**: 6-line addition.

## LOW (1.0-3.9)

### 8. LoginThrottle IP Spoofing Risk

| Metric         | Value                                                                                                                                                          |
| -------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **ID**         | RATE-LIMIT-001                                                                                                                                                 |
| **File**       | `backend/src/auth/services/login-throttle.service.ts:56, auth.controller.ts:297`                                                                               |
| **Risk**       | `@Ip()` decorator reads X-Forwarded-For header without validation. Behind proxy, attacker can spoof IP via `X-Forwarded-For: fake.ip` to bypass rate limiting. |
| **CVSS**       | 3.7 — CVSS:3.1/AV:N/AC:L/PR:N/UI:N/S:U/C:N/I:L/A:N                                                                                                             |
| **Mitigation** | Configure Nestjs trust proxy settings + validate X-Forwarded-For against known proxies.                                                                        |

**Current Status**: Low risk if behind reverse proxy (nginx/ALB), but should
validate trust chain.

**Fix**: In `app.module.ts`, set NestJS to trust only known proxies:

```typescript
app.set('trust proxy', 1); // Trust first proxy (ALB/nginx)
// or validate against known IP list
```

**Effort**: 1-line configuration change.
